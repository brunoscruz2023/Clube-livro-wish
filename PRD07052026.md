## 6.1.3) Constraints e índices (5.3)

### Invariantes críticas
1) Um empréstimo `ACTIVE` por livro  
- índice único parcial: `unique (book_id) where status='ACTIVE'`

2) Renovação no máximo 3  
- `check (renewal_count between 0 and 3)`

3) Livro `AVAILABLE` exige localização consistente  
- `status='AVAILABLE'` => `available_location_type` não null  
- e exatamente um dos IDs preenchido de forma compatível:
  - `HALL` => `available_location_id` NOT NULL e `available_apartment_id` NULL
  - `APARTMENT` => `available_apartment_id` NOT NULL e `available_location_id` NULL

4) Livro `LOANED` não possui campos “available”  
- `status='LOANED'` => `available_location_type`, `available_location_id`, `available_apartment_id` devem ser NULL

### Índices para queries AS-IS
- `books (condo_id, active, title)`
- `book_loans (condo_id, apartment_id, status)`
- opcional: `book_loans (condo_id, borrower_user_id, loaned_at desc)`

## 6.2.2) Estratégia de corte (cutover)
Pré-requisitos:
- export do Firestore antes de qualquer carga
- janela de manutenção comunicada
- AS-IS permanece de pé para rollback

Durante o cutover:
1) Colocar Firestore em modo somente leitura (bloquear writes temporariamente)
2) Rodar ETL full
3) Rodar validação (6.2.3)
4) Se OK: virar tráfego para TO-BE
5) Reabilitar writes no Firestore (ou manter como fallback restrito, conforme política)

## 6.2.3) Validação (pré-corte) — SQL
```sql
-- Contagens (comparar com export Firestore)
SELECT 'condos' AS tabela, COUNT(*) AS total FROM condos
UNION ALL SELECT 'blocks', COUNT(*) FROM blocks
UNION ALL SELECT 'apartments', COUNT(*) FROM apartments
UNION ALL SELECT 'locations', COUNT(*) FROM locations
UNION ALL SELECT 'profiles', COUNT(*) FROM profiles
UNION ALL SELECT 'books', COUNT(*) FROM books
UNION ALL SELECT 'book_loans', COUNT(*) FROM book_loans;

-- No máximo 1 ACTIVE loan por book
SELECT book_id, COUNT(*) AS qtd
FROM book_loans
WHERE status='ACTIVE'
GROUP BY book_id
HAVING COUNT(*) > 1;

-- renewal_count dentro do limite
SELECT id, renewal_count
FROM book_loans
WHERE renewal_count < 0 OR renewal_count > 3;

-- books LOANED sem loan ACTIVE
SELECT b.id
FROM books b
LEFT JOIN book_loans bl ON bl.book_id=b.id AND bl.status='ACTIVE'
WHERE b.status='LOANED' AND bl.id IS NULL;

-- books AVAILABLE com loan ACTIVE
SELECT b.id
FROM books b
JOIN book_loans bl ON bl.book_id=b.id AND bl.status='ACTIVE'
WHERE b.status='AVAILABLE';

-- tenant boundary preenchido
SELECT COUNT(*) FROM books WHERE condo_id IS NULL;
SELECT COUNT(*) FROM book_loans WHERE condo_id IS NULL;
SELECT COUNT(*) FROM profiles WHERE condo_id IS NULL;

-- loans com FK quebrada (deve retornar 0 linhas)
SELECT bl.id
FROM book_loans bl
LEFT JOIN books b ON b.id=bl.book_id
LEFT JOIN apartments a ON a.id=bl.apartment_id
LEFT JOIN profiles p ON p.id=bl.borrower_user_id
WHERE b.id IS NULL OR a.id IS NULL OR p.id IS NULL;
```

## 6.2.4) Rollback
Cenário A — falha antes de virar tráfego:
- não vira tráfego
- reabre writes no Firestore
- corrige ETL
- trunca tabelas no Postgres e reexecuta

Cenário B — falha após virar tráfego:
- reverte app para Firebase (AS-IS)
- reabre writes no Firestore
- registra incidente (causa, duração, correção)
- só tenta novamente com nova janela
